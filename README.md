# STM32_TempHum_sensing

Bare-metal temperature and humidity sensing project using the STM32 Nucleo-F401RE and DHT22 sensor.  

## Hardware


MCU Board-> NUCLEO-F401RE (STM32F401RE, Cortex-M4 @ 84MHz)

Sensor-> DHT22 

Pull-up resistor -> 1kΩ (Data line → 3.3V) 

IDE -> STM32CubeIDE 1.19 

HAL Version -> STM32CubeF4

### Wiring (Arduino headers)


Pin 1 (VDD)      CN6 PIN3 (3.3V)        Power supply

Pin 2 (Data)     CN5 PIN6 (PA6 / D12)   Data line + pull-up 

Pin 3 (NC)               —              Not connected 

Pin 4 (GND)      CN6 PIN7 (GND)         Ground 

*The pull-up resistor (1kΩ) must be connected between the Data pin and 3.3V.  
*The DHT22 uses a single-wire half-duplex protocol — the MCU alternates between output and input mode on the same pin.



## CubeMX Configuration

### Clock
Source: HSI (16MHz internal oscillator)

PLL configured for **HCLK = 84MHz** (maximum for F401RE)

APB1 = 42MHz, APB2 = 84MHz

### USART2 — Virtual COM Port

Mode=Asynchronous 

Baud Rate=115200 

Word Length=8 bit 

Parity=None 

Stop Bits= 1 

USART2 TX/RX are mapped to PA2/PA3, which are internally wired to the ST-Link on the Nucleo board — no extra hardware needed.

### PA6 — DHT22 Data Pin

Mode:GPIO Output Push-Pull 

Pull:No pull-up / No pull-down 

Speed:Very High 

Initial level:High

Pull-up/pull-down is set to None because an external 1kΩ pull-up is used. Speed is set to Very High because the DHT22 protocol requires tight microsecond-level timing.

### IWDG — Independent Watchdog

Clock source:LSI (~32kHz) 

Prescaler: ~1kHz tick (1 tick = 1ms) 

Reload value: 3000 -> 3 second timeout

The DHT22 is sampled every 2 seconds. The IWDG is refreshed before and after each `HAL_Delay(2000)`, giving a comfortable margin. 
If the MCU hangs anywhere for more than 3 seconds, the watchdog resets the system automatically.

### SYS — Timebase

Timebase Source -> TIM1 

TIM1 is used as the HAL timebase instead of SysTick. This frees SysTick for future RTOS integration without conflicts.

### Driver Design

The DHT22 driver is implemented as a standalone module (`dht22.h` / `dht22.c`) following standard embedded C conventions:

- **Public API** declared in `dht22.h` — only two functions exposed
- **Private helpers** (`DWT_Init`, `delay_us`, `pin_output`, `pin_input`, `pin_read`) declared `static` in `dht22.c` — invisible outside the module
- **`DHT22_Status_t` enum** for return codes instead of raw integers
- **`DHT22_Data_t` struct** groups temperature and humidity into a single object passed by pointer

```c
/* Public API */
void           DHT22_Init(void);
DHT22_Status_t DHT22_Read(DHT22_Data_t *data);
```

### Microsecond Timing — DWT Cycle Counter

`HAL_Delay()` has 1ms resolution — too coarse for the DHT22 protocol which requires delays of 15–80μs. Instead, the driver uses the **DWT (Data Watchpoint and Trace) cycle counter**, a hardware register in the Cortex-M4 core that increments every CPU clock cycle.

```c
static void delay_us(uint32_t us)
{
    uint32_t start = DWT->CYCCNT;
    uint32_t ticks = us * (SystemCoreClock / 1000000U); // 84 cycles = 1μs @ 84MHz
    while ((DWT->CYCCNT - start) < ticks);
}
```

### Dynamic Pin Direction

The DHT22 uses a **single wire for both sending and receiving** (half-duplex). The MCU must switch PA6 between output mode (to send the start signal) and input mode (to read the sensor response). This is done by calling `HAL_GPIO_Init()` dynamically during each measurement.

## DHT22 Protocol

The DHT22 uses a proprietary 1-wire protocol. Every measurement follows this sequence:

```
1. MCU pulls line LOW for ≥1ms          → Wake-up signal to sensor
2. MCU releases line HIGH for ~15μs     → End of start signal
3. MCU switches to input mode
4. Sensor pulls LOW ~80μs               → ACK part 1
5. Sensor pulls HIGH ~80μs              → ACK part 2 (ready to send)
6. 40 data bits transmitted:
      Each bit starts with LOW ~50μs
      HIGH 26-28μs → bit '0'
      HIGH 70μs    → bit '1'
7. Bit threshold: high_time > 15 → '1', else '0'
```

### Data Format (40 bits = 5 bytes)

| Byte | Content |
|---|---|
| 0 | Humidity integer part (MSB) |
| 1 | Humidity decimal part (LSB) |
| 2 | Temperature integer part (MSB) + sign bit (bit7) |
| 3 | Temperature decimal part (LSB) |
| 4 | Checksum = byte0 + byte1 + byte2 + byte3 |

### Decoding

```c
uint16_t raw_hum  = ((uint16_t)raw[0] << 8) | raw[1];
uint16_t raw_temp = ((uint16_t)(raw[2] & 0x7F) << 8) | raw[3];

data->humidity    = raw_hum  / 10.0f;   // e.g. 734 → 73.4%
data->temperature = raw_temp / 10.0f;   // e.g. 188 → 18.8°C

if (raw[2] & 0x80)                      // bit7 = sign bit
    data->temperature = -data->temperature;
```

The sensor uses **sign-magnitude** encoding (not two's complement) for negative temperatures — bit7 of byte2 indicates the sign.

### Checksum

```c
uint8_t checksum = raw[0] + raw[1] + raw[2] + raw[3];
if (checksum != raw[4]) return DHT22_CRC_ERR;
```

A simple 8-bit checksum validates data integrity. If a bit is corrupted during transmission (noise, timing jitter, poor contact), the checksum mismatch causes the measurement to be discarded.

## UART printf Redirect

`printf` is redirected to USART2 by overriding the weak `__io_putchar()` function from the standard library:

```c
int __io_putchar(int ch)
{
    HAL_UART_Transmit(&huart2, (uint8_t *)&ch, 1, HAL_MAX_DELAY);
    return ch;
}
```

Float formatting support is enabled by adding `-u _printf_float` to the linker flags (`Project → Properties → C/C++ Build → Settings → MCU GCC Linker → Miscellaneous`).

## Main Loop

```c
while (1)
{
    DHT22_Data_t sensor_data;
    DHT22_Status_t status = DHT22_Read(&sensor_data);

    switch (status) {
        case DHT22_OK:
            printf("Temp: %.1f C  |  Hum: %.1f %%\r\n",
                   sensor_data.temperature, sensor_data.humidity);
            break;
        case DHT22_TIMEOUT:
            printf("ERROR: Timeout\r\n");
            break;
        case DHT22_CRC_ERR:
            printf("ERROR: Checksum\r\n");
            break;
        default:
            printf("ERROR: Unknown\r\n");
            break;
    }

    HAL_IWDG_Refresh(&hiwdg);
    HAL_Delay(2000);            // DHT22 max sample rate: 0.5Hz
    HAL_IWDG_Refresh(&hiwdg);
}
```

The IWDG is refreshed both before and after the 2-second delay to ensure the 3-second window is never exceeded during normal operation.
